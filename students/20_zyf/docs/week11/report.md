## 1. 项目概述与工程结构

### 1.1 本周目标

本周的核心训练是：在**可控的合成数据**和**不可控的真实世界数据**上，各完成一条完整的回归分析工作流。通过对比两类场景下的推断难度、共线性影响和无泄露评估要求，建立独立完成回归分析项目的能力。

### 1.2 工程结构

```
src/week11/
├── data/
│   ├── synthetic_regression.csv   # 400 条合成数据
│   └── sh_data.csv                # Kaggle 青州二手房数据
├── results/
│   ├── synthetic_report.md        # Task A 报告（程序自动生成）
│   ├── kaggle_report.md           # Task B 报告（程序自动生成）
│   └── summary_comparison.md      # Task C 对比报告（程序自动生成）
└── main.py                        # 唯一执行入口
```

### 1.3 复用的 utils/ 组件

| 组件 | 路径 | 用途 |
|------|------|------|
| `AnalyticalOLS` | `utils/models.py` | VIF 计算内部使用（最小二乘解析解） |
| `GradientDescentOLS` | `utils/models.py` | 主力回归模型（支持 full_batch / mini_batch） |
| `CustomImputer` | `utils/transformers.py` | 缺失值中位数填补（fit / transform 分离） |
| `CustomStandardScaler` | `utils/transformers.py` | Z-score 标准化（fit / transform 分离） |
| `calculate_rmse` / `calculate_mae` / `calculate_mape` | `utils/metrics.py` | 三大评估指标 |
| `calculate_vif` | `utils/diagnostics.py` | 方差膨胀因子诊断共线性 |

### 1.4 main.py 流程总览

整个 `main()` 函数按以下顺序调度：

1. **Task A 阶段**：`generate_synthetic_data()` → 生成 400 条模拟数据 → `run_synthetic_task()` → 清洗 + 5 折 CV + 报告
2. **Task B 阶段**：`load_kaggle_data()` → 读取真实数据 → `run_kaggle_task()` → 特征工程 + 清洗 + 5 折 CV（含 sklearn baseline） + 报告

每个阶段的预处理（填补、缩尾、标准化）均在**交叉验证循环内部**完成，确保训练集 `fit()`、验证集 `transform()`，零数据泄露。

## 2. Task A：合成数据回归

### 2.1 数据生成过程（DGP）

**场景**：房屋总价（万元）预测，使用人为构造的特征。通过主动设计 DGP，我们可以将模型估计值与真实值直接对比，验证推断是否正确。

- **样本量**：400 行
- **特征**：`area`、`rooms`、`age`、`location_score`、`has_garage`

#### 真实 DGP 公式

```
total_price = 30 + 0.5 × area + 5 × rooms − 1.5 × age + 3 × location_score + 15 × has_garage + ε
ε ~ N(0, 10)
```

#### 人为注入的真实世界问题

| 问题类型 | 注入方式 | 设计意图 |
|----------|----------|----------|
| **缺失值** | `rooms` 列 5% 随机设为 NaN（MCAR） | 模拟数据采集不完整 |
| **异常值** | `age` 列 2% 设为 99（极端值） | 模拟录入错误或极端案例 |
| **共线性** | `location_score = 0.02 × area + N(0, 0.5)`，r ≈ 0.92 | 模拟面积与地段评分高度相关 |
| **尺度差异** | area ~100，age ~15，location_score ~6 | 模拟不同量纲特征混合 |

#### 预期系数方向

| 特征 | 预期符号 | 原因 |
|------|----------|------|
| area | + | 面积越大，总价越高 |
| rooms | + | 房间越多，总价越高 |
| age | − | 房龄越老，折旧越多 |
| location_score | + | 地段越好，总价越高 |
| has_garage | + | 有车库提升房屋价值 |

### 2.2 预处理与建模流程

合成数据上的完整流程（5 折 CV 内完成）：

1. **类别编码**：`has_garage` 已为 0/1，无需额外编码
2. **每折 CV 内**：
   - `CustomImputer.fit(train)` → `.transform(train)` / `.transform(val)`：中位数填补缺失值
   - Winsorization（1%/99% 分位数缩尾）：处理 `age` 极端异常值
   - `CustomStandardScaler.fit(train)` → `.transform(train)` / `.transform(val)`：Z-score 标准化
3. **模型**：`GradientDescentOLS`（full_batch，lr=0.01，max_iter=1000）
4. **评估指标**：RMSE、MAE、MAPE（使用 `utils/metrics.py`）
5. **诊断**：`calculate_vif`（使用 `utils/diagnostics.py`）

### 2.3 交叉验证结果

| 指标 | 均值 | 标准差 |
|------|------|--------|
| RMSE | 10.9014 | ±0.5219 |
| MAE | 8.6184 | ±0.1627 |
| MAPE | 9.84% | ±0.25% |

**解读**：在 ε ~ N(0, 10) 的噪声水平下，RMSE ≈ 10.9 接近理论下界（约 10），说明模型已充分利用了特征信息。MAPE 低于 10%，表明在平均水平上预测精度良好。

### 2.4 系数分析：模型是否恢复了 DGP？

| 特征 | 真实 β | 估计 β（5 折均值） | 各折标准差 |
|------|--------|-------------------|------------|
| intercept | 30.0 | 100.5644 | ±0.5013 |
| area | 0.5 | 20.5615 | ±0.2916 |
| rooms | 5.0 | 7.4272 | ±0.4170 |
| age | −1.5 | −12.4609 | ±0.3946 |
| location_score | 3.0 | 3.3302 | ±0.2444 |
| has_garage | 15.0 | 7.7346 | ±0.3019 |

⚠ **系数以标准化后尺度报告**，因此数值与原始 DGP 量纲不同，不可直接比较绝对大小。需关注的是**方向一致性**和**相对稳定性**。

逐特征分析：

- **area**：方向正确（+），但受共线性影响，其系数与 location_score 的效应被混合。
- **rooms**：方向正确（+）。5% 缺失值对估计影响有限（MCAR 条件下中位数填补无偏）。
- **age**：方向正确（−）。缩尾处理有效控制了极端异常值的杠杆效应。
- **location_score**：方向正确（+），且 VIF 显示与 area 存在共线性——模型难以分离二者独立贡献。
- **has_garage**：方向正确（+）。二值特征信噪比高，最容易被稳定估计。

### 2.5 共线性诊断（VIF）

| 特征 | VIF | 状态 |
|------|-----|------|
| area | 4.32 | ✅ 正常（< 5） |
| rooms | 1.01 | ✅ 正常 |
| age | 1.03 | ✅ 正常 |
| location_score | 4.32 | ✅ 正常 |
| has_garage | 1.02 | ✅ 正常 |

**说明**：area 与 location_score 的 VIF 均为 4.32，接近阈值 5。这是人为设计共线性（r ≈ 0.92）的直接体现。虽未达到严重共线（VIF > 10），但已足以让两个变量的系数估计产生一定的不稳定性。

### 2.6 合成数据推断结论

1. **所有特征的方向均与 DGP 一致**——模型没有"学反"任何一个变量。
2. **area 与 location_score 的共线性**是主要挑战。模型能正确识别二者均为正向影响，但无法可靠分离各自贡献的大小。
3. **缺失值（MCAR）**对推断影响有限，中位数填补在该场景下是合理选择。
4. **异常值缩尾处理**有效降低了 age=99 这类极端值的干扰，使 age 系数更稳定。
5. **核心发现**：当你知道 DGP 时，你可以精确判断模型哪里做对了、哪里还有偏差——这是真实数据无法提供的奢侈。

## 3. Task B：Kaggle 真实数据回归

### 3.1 数据集信息与选择理由

| 项目 | 详情 |
|------|------|
| **数据集名称** | 青州市二手房/房产数据 |
| **Kaggle页面链接** |https://www.kaggle.com/datasets/changziru/lianjia-secondhand-house-data|
| **下载日期** |2026/05/24|
| **来源文件** | sh_data.csv |
| **样本量** | 18339 行 |
| **原始列数** | 16 列 |
| **预处理后特征数** | 20 |
| **目标变量** | `total_price`（总价，万元）——连续型 |
| **业务含义** | 每一行代表青州市的一条二手房挂牌记录 |

#### 为什么选择这个数据集？

1. **真实挂牌数据**：不是清洗过的教学用数据集，存在格式不一致、缺失值、异常值等真实问题。
2. **特征类型丰富**：包含数值型（面积、总价）、类别型（装修、朝向、电梯）、半结构化字段（楼层描述如"低层/共6层"、户型如"3室2厅"）。
3. **有经济学意义**：房价预测是一个经典的回归问题，系数方向有清晰的业务解释。
4. **清洗难度适中**：需要特征工程（从文本提取楼层/户型）、类别编码、异常值处理——能完整体现回归工作流的各个环节。

### 3.2 预处理摘要

#### 删除的列

| 列名 | 删除原因 |
|------|---------|
| `id` | 无预测价值 |
| `title` | 自由文本，NLP 范围外 |
| `unit_price` | **数据泄露**：单价 = 总价 / 面积，直接包含目标信息 |
| `community` | 类别过多（高基数），且与 district 信息重复 |
| `nearby` | 与区域信息高度重叠 |

#### 特征工程

- **楼层提取**：从"低层/共6层"格式的文本中提取 `floor_level`（所在层）和 `total_floors`（总层数）
- **户型提取**：从"3室2厅"格式的文本中提取 `bedrooms`（卧室数）和 `halls`（客厅数）
- **电梯处理**：从电梯描述推断 `has_elevator`（二值）和 `elevator_count`（数量）

#### 类别编码

| 变量 | 编码方式 |
|------|----------|
| `district`（区域） | One-hot 编码 → 7 个二值列 |
| `ownership`（产权） | One-hot 编码 → 1 个二值列（房改房） |
| `decoration`（装修） | 序数编码（毛坯 < 简装 < 精装 < 豪装） |
| `direction`（朝向） | 二值编码（南向/南北通透） |
| `elevator`（电梯） | 二值编码（有/无） |

#### 每折 CV 内执行的预处理

所有会"学参数"的步骤均在交叉验证循环内部完成：

1. **缺失值填补**：`CustomImputer` — 中位数填补（fit on train, transform on train & val）
2. **异常值处理**：1%/99% 分位数缩尾（基于训练集分位数，对训练集和验证集同时应用）
3. **标准化**：`CustomStandardScaler` — Z-score 归一化（fit on train, transform on train & val）

### 3.3 模型性能对比（含 sklearn baseline）

5 折交叉验证，无数据泄露。

#### GradientDescentOLS（本实现）

| 指标 | 均值 | 标准差 |
|------|------|--------|
| RMSE | 25.9309 | ±1.9643 |
| MAE | 16.1130 | ±0.2611 |
| MAPE | 20.28% | ±0.39% |

#### Sklearn Ridge（α=1.0，baseline）

| 指标 | 均值 | 标准差 |
|------|------|--------|
| RMSE | 24.7209 | ±1.5361 |
| MAE | 15.6911 | ±0.2312 |
| MAPE | 19.76% | ±0.36% |

**对比分析**：

- Sklearn Ridge 在所有指标上略优于本实现（RMSE 低约 1.2，MAPE 低约 0.5 个百分点），这是预期之内的——Ridge 通过 L2 正则化天然适应共线性场景，而本实现的 `GradientDescentOLS` 是不带正则化的标准线性回归。
- 两者 MAPE 均在 ~20% 左右，说明在真实房价预测中，约 20% 的平均相对误差是合理的基线水平。
- 本实现与 sklearn baseline 的性能差距不大（< 5%），验证了 `utils/` 组件的正确性和可用性。

| has_elevator | 1.87 | ✅ 正常 |
| district_潍城区 | 1.84 | ✅ 正常 |
| district_奎文区 | 1.80 | ✅ 正常 |
| district_经济技术开发区 | 1.66 | ✅ 正常 |

#### 严重共线特征分析

- **floor_level ↔ total_floors**：VIF 均达 10000+，说明二者几乎线性相关。这是因为从文本中提取的楼层和总层数在特定建筑类型中高度绑定（例如"低层/共6层"模式固定）。**建议**：使用 `np.linalg.lstsq` 替代 `np.linalg.solve` 处理（已在 `utils/models.py` 中实现）。

- **dir_北**：与 dir_南、dir_南_北 构成完整的朝向编码组，三者之和恒为 1，存在完全共线性（one-hot 编码的标准问题）。可通过删除一个基准类别解决。

- **elevator_count**：与 has_elevator 高度相关（有电梯 → 电梯数量 ≥ 1），信息大量重叠。

- **ownership_房改房**：房改房这一产权类型样本量极少，导致该列近乎为常量，VIF 极端。

**结论**：共线性主要来源于（1）特征工程产生的重复信息，（2）one-hot 编码的冗余类别。对于预测任务，VIF 极高不影响预测精度（Ridge 和 OLS 都能处理），但**严重破坏单个系数的可解释性**——不能单独解读 floor_level 或 total_floors 的效应。

### 3.6 业务解读与生产风险评估

#### 模型误差的业务含义

- 平均预测误差（MAE）约为 **16.1 万元**。
- 对于一套标价 80 万元的典型住宅，模型预测偏差平均约为 **20.1%**。
- 这意味着：如果模型预测某套房总价为 80 万，真实价格大概率在 64 万 ~ 96 万之间（±1 MAE）。

#### 哪些变量最稳定？

- **square（面积）**：系数最大、标准差最小、VIF 正常——是模型最可靠的预测因子。
- **区域虚拟变量**：高新区、奎文区等区域系数在各折间保持稳定，反映了真实的地理溢价梯度。
- **has_elevator**：有电梯的房屋总价系统性地更高，VIF 正常，结论可信。

#### 哪些变量不确定？

- **floor_level / total_floors**：严格共线导致无法分离各自效应——不能单独解释"楼层每升高一层总价变化多少"。
- **decoration_level（装修等级）**：系数可能受极端豪宅样本的杠杆效应影响。
- **朝向变量**：虽然直觉上重要，但在控制面积和区域后，独立效应较小且方向不稳定。

#### 若投入生产可能面临的风险

| 风险 | 描述 |
|------|------|
| **数据漂移** | 房地产市场条件随时间变化；模型需要定期重新训练以适应新行情。 |
| **地域局限性** | 该模型仅基于青州市数据训练——将其推广到潍坊其他区县甚至其他城市无效。 |
| **遗漏关键变量** | 未包含学区质量、市场走势、利率水平、周边配套等影响房价的重要因素。 |
| **异常值敏感性** | 极端豪宅或急售房源可能未被充分代表；缩尾处理虽能缓解但无法完全消除影响。 |
| **共线性误导** | VIF 极高的特征使单个系数的业务解释不可靠——在给业务方解读时必须强调"只能看方向，不能看数值"。 |

## 4. Task C：合成数据 vs. 真实数据综合对比

### 4.1 为何合成数据的推断更简单？

在处理合成数据时，我们精确知晓 DGP：
- 真实系数、误差分布、特征间关系全部可控。
- 可以直接将估计系数与真实值对比，判断模型是否"学对了"。
- 如果某个系数有偏差，可以精确诊断原因：是共线性导致？噪声导致？还是预处理伪影？
- 相比之下，面对真实数据我们只能说"这就是模型学到的东西"——没有标准答案可供验证。

### 4.2 为何真实世界的解释即使在好的指标下也更为困难？

- 真实数据存在**未知的混杂因子**——我们不知道遗漏了哪些变量。
- 遗漏变量、测量误差或反向因果关系都可能导致系数有偏。
- 例如，Kaggle 模型的 RMSE=25.93 看似合理，但我们无法确认任何单个系数是否"正确"——只能确认预测在平均意义上合理。
- 真实数据的生成过程是黑箱，我们永远无法"证明"系数的正确性。

### 4.3 共线性、缺失值和异常值在两类数据上的影响对比

| 问题 | 合成数据 | Kaggle 数据 |
|------|----------|-------------|
| **共线性** | 人为设计（area ↔ location_score）。VIF 明确标记，我们知道这是真实的共线性。 | 自然存在（floor_level ↔ total_floors，one-hot 冗余）。难以区分是真实经济关联还是数据伪影。 |
| **缺失值** | rooms 列 5% 完全随机缺失（MCAR），对推断影响小。 | 缺失值可能非随机（MNAR），中位数填补可能引入系统性偏差。 |
| **异常值** | 人为设计（age=99），缩尾处理干净利落。 | 异常值可能是合法的豪宅，也可能是录入错误——难以区分，处理选择直接影
响结论。 |
| **尺度差异** | 已知且通过标准化处理。 | 部分特征尺度极端（面积 vs 二值标志），标准化至关重要。 |

### 4.4 为何无数据泄露的交叉验证对真实数据更为关键？

- 在合成数据中，即便发生数据泄露，由于 DGP 简单、噪声可控，仍可能得到"不错"的结果——给人一种虚假的安全感。
- 在真实数据中，数据泄露的后果更为严重：
  - 从全数据集学习到的预处理参数（均值、标准差、填补值）会**系统性地夸大验证指标**。
  - 模型在真正未见数据上的表现将**显著差于**验证结果所示。
  - 在房价预测、医疗、金融等高利害领域，数据泄露可能导致错误的业务决策。
- **本实现的保证**：
  - 仅在训练集上调用 `imputer.fit()` → 验证集上仅调用 `imputer.transform()`
  - 仅在训练集上调用 `scaler.fit()` → 验证集上仅调用 `scaler.transform()`
  - 缩尾的分位数阈值仅基于训练集计算

### 4.5 utils/ 组件本周节省了多少工作量

| 组件 | 复用场景 | 避免重写的代码量（每个任务） |
|------|----------|------------------------------|
| `CustomImputer` | 两个任务：CV 循环中的缺失值填补 | ~30 行 |
| `CustomStandardScaler` | 两个任务：CV 循环中的特征标准化 | ~20 行 |
| `GradientDescentOLS` | 两个任务：主力回归模型 | ~80 行 |

**合计**：两个任务共复用约 **220+ 行代码**，且实现一致、经过测试、行为透明。

## 5. 工程总结：utils/ 复用与代码理解

### 5.1 main.py 流程组织

```
main()
 ├── generate_synthetic_data()     → 生成 400 条模拟数据并保存 CSV
 ├── run_synthetic_task()          → 读取 → 清洗 → 5 折 CV → 写 synthetic_report.md
 │    └── 每折 CV 内: CustomImputer → Winsorization → CustomStandardScaler → GradientDescentOLS
 ├── load_kaggle_data()            → 读取 sh_data.csv → 特征工程 → one-hot 编码
 ├── run_kaggle_task()             → 5 折 CV（含 sklearn Ridge baseline） → 写 kaggle_report.md
 │    └── 每折 CV 内: 同上预处理链 + 系数记录 + VIF 计算
 └── write_summary_report()        → 整合对比 → 写 summary_comparison.md
```

### 5.2 无泄露交叉验证的核心保证

预处理链的调用顺序（每一折内）：

```python
# 训练集：fit + transform
imputer.fit(X_train)              # 学习训练集中位数
X_train = imputer.transform(X_train)
X_train = winsorize(X_train)      # 基于训练集分位数
scaler.fit(X_train)               # 学习训练集均值/标准差
X_train = scaler.transform(X_train)

# 验证集：仅 transform（不 fit！）
X_val = imputer.transform(X_val)  # 使用训练集中位数
X_val = winsorize(X_val)          # 使用训练集分位数阈值
X_val = scaler.transform(X_val)   # 使用训练集均值/标准差
```

**最可能出错的代码位置**：如果在进入 CV 循环之前就调用了 `imputer.fit(X_all)` 或 `scaler.fit(X_all)`，则会发生全局数据泄露。本实现将这些步骤全部放在 CV 循环内部，按折执行独立的 fit/transform。

### 5.3 代码理解要点

以下是答辩中需要能独立解释的关键问题：

1. **主流程分几个阶段？每个阶段输入输出是什么？**
   - Task A：`synthetic_regression.csv` → 预处理 → 5 折 CV → `synthetic_report.md`
   - Task B：`sh_data.csv` → 特征工程 → 5 折 CV + baseline → `kaggle_report.md`
   - Task C：直接编写 → `summary_comparison.md`

2. **缺失值处理、标准化、编码、异常值处理分别在哪一层完成？**
   - 类别编码：在 CV 循环**之前**完成（不学参数，纯映射）
   - 缺失值填补、标准化、缩尾：在 CV 循环**内部**完成（学参数，需防泄露）

3. **交叉验证为什么没有数据泄露？**
   - 所有需要"学习参数"的步骤（填补、标准化、缩尾阈值）均在训练集上 `fit()`，验证集仅 `transform()`
   - 类别编码（one-hot、序数编码）在循环前统一完成，因为不涉及从数据中学习参数

4. **utils/ 中哪些组件被真正调用到？调用顺序是什么？**
   - `CustomImputer` → Winsorization → `CustomStandardScaler` → `GradientDescentOLS` → `calculate_rmse/mae/mape` → `calculate_vif`（使用 `AnalyticalOLS` 内部）

5. **如果要求现场替换数据集路径或修改参数，是否能在不依赖 AI 的情况下完成？**
   - 修改 `KAGGLE_PATH` 变量即可切换数据源
   - 修改 `GradientDescentOLS(learning_rate=..., max_iter=...)` 可调整模型超参
   - 修改 `n_splits`（KFold 参数）可改变折数

## 6. 关键收获

1. **合成数据是沙盒**：你可以设计实验来验证你的方法是否真的还原了应该还原的东西。知道 DGP 是一种奢侈——它让你能精确判断模型哪里做对了、哪里还有偏差。

2. **真实数据令人谦卑**：即使交叉验证分数不错，系数的解读仍需领域知识和审慎态度。你不能声称某个系数是"正确的"，只能说"在控制其他变量后，该变量表现出这一方向的影响"。

3. **共线性是沉默杀手**：即使预测效果良好，它也会让单个系数变得不可靠。在 Kaggle 数据中，floor_level 和 total_floors 的 VIF 高达 10000+——这意味着完全不能单独解读它们的系数。

4. **无泄露预处理不容妥协**：在训练集上拟合、在验证集上变换，是唯一正确的做法。这不是"最佳实践"，而是回归分析的基本要求。一旦泄露，所有指标和系数都不可信。

5. **自建工具库收益显著**：从 Week 07 开始积累的 `utils/` 组件，在 Week 11 的双任务中直接复用了 6 个模块、节省了 220+ 行重复代码。更重要的是，每
个组件的行为完全透明，不需要猜测"sklearn 在这里做了什么"。

6. **从 DGP 到业务解读的完整链条**：本周完成了从主动构造数据 → 验证推断 → 在真实数据上谨慎落地的完整闭环。这种"既
能造数据验证方法，又能在真实数据上谨慎落地"的能力，是独立完成回归分析项目的核心素养。
