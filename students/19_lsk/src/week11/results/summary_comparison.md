# 🧠 Task C: Synthetic vs Real — Summary Comparison

## 1. 为什么模拟数据中"推测"相对容易?

在模拟数据中, 我们**明确知道 DGP (Data Generating Process)**——
包括每个变量的真实系数、噪声分布、共线性结构和异常值位置。
这使得我们可以:
- 直接比较估计系数与真实系数;
- 精确判断模型偏差来自哪里 (噪声? 共线性? 异常值?);
- 主动设计实验: "如果我的清洗流程正确, 参数应该恢复到什么程度?"

简而言之, 模拟数据给了我们一个"标准答案", 推测=对答案。

## 2. 为什么真实数据中即使分数还可以, 解释也更困难?

真实数据 (如 California Housing) 中:
- **没有标准答案**: 我们不知道真正的系数是多少;
- **遗漏变量偏差 (OVB)**: 重要变量 (利率、学区、政策) 可能完全缺失;
- **反向因果**: 高房价可能反过来吸引高收入人群, 系数不是单向的;
- **测量误差**: 人口普查数据本身有抽样和汇总误差;
- **混杂因素**: 多个变量同时影响 X 和 Y, 而我们无法完全控制。

所以, 即使 R² = 0.6 (看起来不错), 我们也不能轻易说"X 导致 Y",
只能谨慎地说"在控制其他变量后, X 与 Y 存在统计关联"。

## 3. 共线性、缺失值、异常值在两类数据上的影响

| 问题 | 模拟数据 | 真实数据 |
|------|----------|----------|
| **共线性** | 我们主动构造, 知道哪两个变量相关。清理时可以从容处理。 | 共线性是"意外发现", 可能需要多次探索才知道哪些变量相关。 |
| **缺失值** | 我们控制缺失机制 (MCAR), 填充效果好。 | 缺失机制未知 (可能是 MAR 或 MNAR), 简单填充可能引入偏差。 |
| **异常值** | 我们知道哪些是异常值 (因为我们注入的), 可以精确评估处理效果。 | 不知道"正常"边界在哪, 可能误将重要信号当作异常值删除。 |

## 4. 为什么"无泄露交叉验证"在真实数据上尤其重要?

真实数据中, 无泄露 CV 是**防止自欺欺人**的最后防线:

1. **预处理泄露**: 如果用全量数据 fit scaler, 验证集信息"泄露"进了训练过程,
   CV 分数会被高估, 但上线后表现会差很多。
2. **缺失值填充泄露**: 用全量数据的中位数填充 → 训练集"看到了"验证集的信息。
3. **特征选择泄露**: 基于全量数据选择特征 → 选中的特征在 CV 中看起来总是好的。

模拟数据中我们至少知道"真实模型"是什么, 可以交叉验证。
真实数据中, CV 分数往往是我们唯一可信的性能指标。
如果 CV 有泄露, 整个评估体系就崩塌了。

## 5. 自己维护的 `utils/` 组件帮助省下了哪些重复劳动?

这周 `utils/` 组件让我省下了以下重复劳动:

| 组件 | 省下的工作 |
|------|------------|
| `models.py` | 不需要每次手写 OLS 正规方程或梯度下降; AnalyticalOLS 自带 summary 和 F-test |
| `metrics.py` | RMSE/MAE/MAPE 一行调用, 不用每次写 numpy 公式 |
| `transformers.py` | Scaler/Imputer 有统一的 fit/transform 接口, 在 CV 循环中无泄露使用 |
| `diagnostics.py` | VIF 计算 + 格式化报告一键生成, 不用手动做回归 |

核心价值: **接口一致性**。所有组件遵循 `fit()` → `transform()` / `predict()` 模式,
使得在交叉验证中无缝切换预处理和模型步骤, 避免因接口不一致导致的泄露风险。
