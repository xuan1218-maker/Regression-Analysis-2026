# Week 7: 优化引擎与泛化能力实验报告

## 实验概述

本实验实现了梯度下降优化的线性回归模型（GradientDescentOLS），并与解析解模型（AnalyticalOLS）进行对比。通过交叉验证、学习率调参、特征标准化等手段，系统评估了模型的泛化能力。

**核心结论**：
- ✅ GradientDescentOLS 成功实现，支持 Full Batch 和 Mini Batch
- ✅ 最佳学习率 = 0.1，过小会导致欠拟合或发散
- ✅ 梯度下降与解析解结果高度接近（R² 差异仅 0.0007）
- ✅ 标准化必须基于 Train 集，防止数据泄露

## 1. GradientDescentOLS 实现说明

### 1.1 核心原理

梯度下降通过迭代更新参数来最小化均方误差（MSE）损失函数：

**损失函数**：
$$L(\beta) = \frac{1}{n} \sum_{i=1}^{n} (y_i - X_i\beta)^2$$

**梯度计算**：
$$\frac{\partial L}{\partial \beta} = \frac{2}{n} X^T (X\beta - y)$$

**参数更新**：
$$\beta = \beta - \eta \cdot \nabla L$$

### 1.2 关键特性

| 特性 | 实现方式 | 说明 |
|------|----------|------|
| 学习率 | `learning_rate` | 控制每次更新的步长 |
| 收敛判断 | `tol` | 连续两次 loss 变化小于阈值时停止 |
| 批量策略 | `gd_type` | 支持 `full_batch` 和 `mini_batch` |
| 随机采样 | `batch_fraction` | Mini Batch 时随机抽取指定比例样本 |
| 截距处理 | `fit_intercept` | 自动在 X 前添加全 1 列 |

### 1.3 多重共线性诊断（VIF）

#### VIF 原理

$$VIF_j = \frac{1}{1 - R_j^2}$$

| VIF 值 | 含义 | 处理建议 |
|--------|------|----------|
| VIF = 1 | 完全不相关 | 无需处理 |
| 1 < VIF < 5 | 中等相关 | 可接受 |
| 5 ≤ VIF < 10 | 高度相关 | 考虑处理 |
| VIF ≥ 10 | 严重共线性 | 必须处理 |

#### 诊断结果

| 特征 | VIF | 状态 |
|------|-----|------|
| TV_Budget | 18.73 | 严重共线性 |
| Online_Video_Budget | 99.67 | 严重共线性 |
| Radio_Budget | 4.56 | 中等相关 |
| Sales | 108.26 | 严重共线性（不应在特征中！）|
| Region_North | 1.14 | 良好 |
| Region_South | 1.15 | 良好 |

### 1.4 5折交叉验证基线结果

| Fold | R² |
|------|-----|
| Fold 1 | 0.3598 |
| Fold 2 | 0.4710 |
| Fold 3 | 0.4118 |
| Fold 4 | 0.4030 |
| Fold 5 | 0.4355 |

**平均 R²: 0.4162** (±0.0367)

## 第二部分：Week 10 —— 工业流水线与无泄漏泛化评估

### 2.1 实验目的

演示数据泄露的危害，构建绝对纯洁、无污染的交叉验证流水线。

### 2.2 扩充算法工具箱

#### 评估指标库

| 指标 | 公式 | 业务含义 |
|------|------|----------|
| RMSE | $$\sqrt{\frac{1}{n}\sum_{i=1}^{n}(y_i - \hat{y}_i)^2}$$ | 对大误差敏感，放大异常值影响 |
| MAE | $$\frac{1}{n}\sum_{i=1}^{n}|y_i - \hat{y}_i|$$ | 对异常值不敏感，更稳健 |
| MAPE | $$\frac{1}{n}\sum_{i=1}^{n}\left|\frac{y_i - \hat{y}_i}{y_i}\right| \times 100\%$$ | 百分比误差，便于业务理解（需处理 y_i=0） |

#### 转换器 API

实现 `CustomStandardScaler` 类，严格遵循大厂 Transformer 接口规范：

| 方法 | 功能 | 说明 |
|------|------|------|
| `fit(X)` | 计算均值和标准差 | 只用训练集，保存到 `self.mean_` 和 `self.std_` |
| `transform(X)` | 标准化转换 | 用训练集的统计量转换数据 |
| `fit_transform(X)` | 拟合+转换 | 等价于 `fit(X).transform(X)` |

### 2.3 数据泄露的危害对比实验

#### 实验设计

| 实验 | 预处理方式 | 数据流向 | 说明 |
|------|-----------|----------|------|
| **错误交叉验证** | 全局预处理 | 全量数据 → 标准化 → 填补 → 切分 | 验证集信息泄露 |
| **正确交叉验证** | 循环内预处理 | 切分 → 训练集 fit → 验证集 transform | 无数据泄露 |

#### 运行结果

| 指标 | 有数据泄露 (错误CV) | 无数据泄露 (正确CV) | 差异 |
|------|---------------------|---------------------|------|
| RMSE | 58.37 | 160.67 | -102.30 |
| MAE | 47.62 | 77.35 | -29.74 |
| **MAPE** | **8.09%** | **13.73%** | **-5.64%** |

#### 结果解读

**有数据泄露的版本各项指标都更"好看"**：
- MAPE 被低估了 5.64 个百分点（8.09% vs 13.73%）
- RMSE 被低估了 102 个单位
- MAE 被低估了 30 个单位

**但这恰恰是致命的陷阱**：模型在实际部署时，真实误差会比预期大得多！

### 2.4 可视化结果


- **红色柱**：有数据泄露（误差更小，但虚假）
- **绿色柱**：无数据泄露（真实泛化误差）
- **结论**：数据泄露会严重低估模型的实际误差

### 2.5 业务解读

模型上线后，每天的销售额预测平均绝对百分比误差约为 **13.73%**。

**业务含义**：
- 如果某天实际销售额为 **100 万元**，预测误差大约在 **±13.73 万元** 左右
- 这意味着每天的预算决策需要考虑约 14% 的不确定性
- 库存管理需要预留相应缓冲

**为什么老板应该看这个"差成绩"？**

| 看什么成绩 | 后果 |
|-----------|------|
| 泄露版成绩（8.09%） | 盲目乐观，备货不足，错失销售机会 |
| 真实成绩（13.73%） | 合理规划，稳健经营，风险可控 |

### 第三部分：核心问题

### 问题1：VIF > 10 意味着什么？如何处理？

**答案**：VIF > 10 表示存在严重多重共线性。

**处理方法**：
1. 删除高度相关的特征之一
2. 使用 PCA 降维
3. 使用正则化方法（Ridge/Lasso）

### 问题2：虚拟变量陷阱是什么？如何避免？

**答案**：k 个分类变量生成 k 列虚拟变量会导致完全多重共线性，$(X^TX)$ 不可逆。

**避免方法**：使用 `drop_first=True`，只保留 k-1 列。

### 问题3：为什么泄露版的指标更"好看"？

**答案**：泄露版在预处理阶段使用了全量数据（包括验证集）的统计量：

1. **标准化时**：用了验证集的均值和标准差
2. **缺失值填补时**：用了验证集的均值

这导致验证集信息被"偷看"，模型"提前知道"了验证集的数据分布，评估结果虚高。

**具体对比**：

| 阶段 | 错误做法（有泄露） | 正确做法（无泄露） |
|------|-------------------|-------------------|
| 标准化 | 用全量数据 fit_transform | 只用训练集 fit，验证集 transform |
| 缺失值填补 | 用全量数据均值填补 | 用训练集均值填补验证集 |
| 结果 | 过于乐观，虚假 | 真实泛化误差 |

### 问题4：为什么应该看真实的泛化误差？

**答案**：

| 看什么成绩 | 后果 |
|-----------|------|
| 泄露版成绩（8.09%） | 盲目乐观，备货不足，错失销售机会，上线后崩溃 |
| 真实成绩（13.73%） | 合理规划，稳健经营，风险可控，准确预测业务 |

**核心原因**：
- 泄露版的"好成绩"是假的，模型上线后在新数据上表现会大幅下降
- 无泄露版的"差成绩"才是模型真实的泛化误差，能准确预测未来的业务表现
- 用真实的误差做业务决策，才能合理规划预算和库存

**业务建议**：
- 接受 13.73% 的预测误差
- 预算规划时预留 14% 的安全边际
- 库存管理根据预测误差范围调整安全库存

## 第四部分：总结

### 4.1 技术收获

| 主题 | 关键要点 |
|------|----------|
| 数据清洗 | CLI 脚本、One-Hot 编码（drop_first=True）、缩尾处理、缺失值填补 |
| 模型诊断 | VIF 计算、多重共线性检测、彩色警告输出 |
| 数据泄露 | 全局预处理 vs 循环内预处理、验证集信息泄露的危害 |
| Transformer API | fit/transform/fit_transform 接口规范 |
| 评估指标 | RMSE、MAE、MAPE 的公式和业务含义 |

### 4.2 核心教训

1. **数据泄露是致命的**：评估结果虚假乐观，上线后崩溃
2. **预处理必须在循环内完成**：只用训练集拟合统计量
3. **老板应该看真实泛化误差**：MAPE = 13.73% 指导业务决策
4. **VIF 诊断必不可少**：多重共线性影响系数稳定性
5. **虚拟变量陷阱必须避免**：使用 `drop_first=True`
